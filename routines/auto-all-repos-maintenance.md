# Fleet daily maintenance routine (the single scheduled entry point)

A portable, **project-agnostic** spec for the **one** daily routine that maintains **every repo the owner has opted into Claudinite** — scheduled once, from the canon repo, and reaching all the others. It carries no maintenance logic of its own. It is a thin **orchestrator** that does two things: **(1)** run the **planner** logic — a pack-agnostic walk over the repos it was handed that decides *what work each repo needs* in code and emits the plan — then **(2)** dispatch each planned unit as an isolated worker **at the tier its `smarts` names**. Every unit is independent — they run concurrently, nothing waits on anything else. The point is twofold: the owner registers **a single schedule** for **all** their repos, and every unit is **guaranteed to run** because each is its own isolated subagent (or code action) — one failing, stalling, or exiting early cannot stop the others. The "should I run" decision is **code** (the planner's gates), never this routine's judgment.

Two deterministic **migration passes** bracket that pack work: **before** any pack runs, the migration **apply pass** brings every covered member to the current canonical shape; **after** every unit settles, the migration **retire pass** finalizes (retires) any migration the fleet has fully applied and left quiet for a cycle. Both are pure decision logic (`smarts: none`) the orchestrator runs directly: the orchestrator **passes them this session's scoped repos** and the MCP-backed `gh` reader/writer, and they act only on those repos. The migrations flow owns them ([../migrations/fleet-apply.mjs](../migrations/fleet-apply.mjs), [../migrations/fleet-retire.mjs](../migrations/fleet-retire.mjs)); the routine only sequences them around the plan, adding no reasoning of its own. Retirement targets the **canon repo the migrations live in** — the same place they are loaded from, not a separately-discovered "home".

This routine **replaces** scheduling anything individually **and** any per-repo maintenance schedule: schedule *this* one, in the canon repo, and nothing else. Do **not** also schedule the planner, a pack's task, or a per-repo routine. The planner and each step's spec stay exactly as written — this routine only dispatches the plan the planner emits, concurrently, in isolation.

## Conventions used in this doc

- **Canon repo.** This routine is scheduled from, keeps its tracking issue in, and ships its migrations from a single fixed **canon repo** — the repo where you vendor and schedule this doc (for the reference deployment, that's Claudinite itself). "Canon repo" below means that repo; every *other* repo it maintains is a **member repo**. It is a known constant the orchestrator names — nothing is discovered from a CI env var.
- **Default branch.** Each member repo's own default branch is whatever *that* repo uses; the plan and workers substitute it per repo, so this routine never assumes `main` for a member repo.
- **GitHub access is MCP-native — no REST client, no token.** This routine, its `smarts: none` code steps, and its workers all reach GitHub **only** through the session's **GitHub MCP tools** (`mcp__github__*`). The orchestrator knows the repos it manages from **this session's granted scope** — it does not enumerate GitHub — and backs the code steps' injected I/O with those MCP tools. There is no `GITHUB_TOKEN`, no `gh`/`curl`, and no raw `fetch` to `api.github.com` anywhere in this process: in a scheduled web session the shell has no GitHub REST credential, and cross-repo `git` checkout is impossible, so MCP is the whole surface. Two injected shapes, both the orchestrator's to fulfill over MCP:
  - The **planner** reads through a read-only `gh(path) -> { status, json }` — each path a `list_*` / `get_file_contents` call.
  - The **migration passes** take a semantic **`io`** object whose methods map **1:1 to MCP write/read tools**, so no Git-Data-API shapes leak into the code: `io.getDefaultBranch` (get repo), `io.read` (`get_file_contents`), `io.ensureBranch` (`create_branch`), `io.commit` (`push_files`), `io.remove` (`delete_file`), `io.hasOpenPr` (`list_pull_requests`), `io.openPr` (`create_pull_request`), `io.openIssue` (`issue_write`). Writes land as one `push_files` commit; a rare file-move's delete follows via `delete_file` (MCP has no atomic multi-file+delete — no active migration produces a delete).

## What the planner decides, what this routine dispatches

The **planner** ([fleet/DESIGN.md](fleet/DESIGN.md), [fleet/plan.mjs](fleet/plan.mjs)) is **pack-agnostic decision logic**: it goes over the repos it was handed — every covered member, plus the **canon repo itself, planned last** with the fleet aggregate its home-only packs' gates need — and for each assembles the maintenance actions its declared packs contribute (their `run_daily` tasks), runs each action's **"should I run" gate** (pure code), and emits the day's **work plan** — a flat list of `(repo, task, worker, targets, reason, order, smarts)` **units**, one per `(repo, task)` a gate marked live. A repo with nothing to do yields no units, at an MCP-read's cost. **The planner never dispatches or depends on any one pack's workflow** — every unit stands alone, so an un-set-up or failing pack surfaces as one isolated unit (a baselining fix), never a reason the plan can't be built. This routine runs the planner over the repos it was handed, reads its units, and dispatches them; it decides *nothing* itself.

The units span every kind of maintenance, each deferring wholly to its own doc:

- **Growth** (the growth pack owns the lifecycle narrative) — the member-side tasks (`growth-extract-new-instructions`, `growth-dedup-local-instructions`, `growth-discover-packs`) on every repo declaring the growth pack, and the central `growth-promote-to-claudinite` on the canon repo via its home-only curation pack. **All ordinary, independent planned units** — no phases, no barriers, no bespoke central step: each stage reads only what's already merged, so extract→promote→dedup propagates across successive nightly runs (and promote's PR approval was always the dominant latency).
- **The repo-tidying pack** — the single `repo-tidy` task (assess branches/PRs read-only, act on issues, reconcile the standing tracker, in one pass) on every repo declaring the pack, and against the canon repo.
- **`baselining`** (the baseline pack's task — the member re-run of the idempotent bootstrap + check-alignment); a handed repo that isn't covered yet gets its **first baseline** (adoption) from the same worker.
- **Any pack task** — e.g. `chrome-store-release`. New maintenance is a new pack `(gate, worker)` pair; the plan picks it up automatically — **no edit here**.

## Step 0 — apply migrations across the fleet (before any pack work)

Before the planner runs, bring every managed repo to the current canonical shape: run the migration **apply pass** ([../migrations/fleet-apply.mjs](../migrations/fleet-apply.mjs))'s `applyToRepos(io, repos, migrations)` over this session's scoped repos, with the MCP-backed `io` object and the migration set (loaded via `migrations/registry.mjs`). It computes each member's pending migration writes — the declared file renames, template materializations, and ref rewrites — and lands them as **one commit** through `io.commit` (`push_files`), honoring the member's `push`/`pr` delivery. It is **pure decision logic (`smarts: none`** — no subagent) and idempotent: a member already on the canonical shape gets no commit. Running migrations *first* means the pack tasks (baselining's check-alignment included) evaluate the already-migrated shape. It **returns the ids it applied this cycle** — hold that set in hand and pass it to **Step 3** as its quiescence evidence (no intermediate file).

## Step 1 — run the planner over the handed repos

The orchestrator already knows the repos it manages — **this session's granted scope** — so run the **planner** ([fleet/plan.mjs](fleet/plan.mjs))'s `planGivenRepos(gh, canonRepo, repos)` over them with the MCP-backed `gh`, and read the returned plan. For each repo it assembles the declared packs' `run_daily` tasks ([fleet/registry.mjs](fleet/registry.mjs)) and runs each task's **gate** ([fleet/gates.mjs](fleet/gates.mjs) — pure "should I run" code) over the repo's signals ([fleet/signals.mjs](fleet/signals.mjs)). The result is the unit list — `buildWorkPlan` *is* exactly this loop over the repos you hand it. **The worklist is decided by code, not by this routine's reasoning.**

**Local-pack tasks — experimental, not enabled by default.** A member's canon-pack tasks are on disk in the canon checkout; its **own local packs** (`.claudinite/local_packs/`) could also contribute `run_daily` tasks, read from the member repo by the `localTasksFor` seam ([fleet/local-tasks.mjs](fleet/local-tasks.mjs)'s `readLocalTasks(gh, fullName)`). That path is built and tested but **not yet proven for the load and variety of arbitrary member-authored daily jobs**, so **do not pass `localTasksFor`** — `planGivenRepos(gh, canonRepo, repos)` with the seam unset plans **no** local-pack daily tasks (a project's scheduled work stays a canon-pack task or an out-of-repo routine for now). Enable it deliberately — pass `{ localTasksFor }` — only once the mechanism is proven; the reader's header states the single-owner-trust contract it runs under.

If the planner logic itself **fails** (the gate code errors across the board), or its `errors` show probes that couldn't cover repos you *can* reach, you **cannot guarantee coverage** — log it (see Tracking) and dispatch whatever plan you have. But a **broken or un-set-up pack is not that failure**: it surfaces as an isolated adoption/baselining unit for that one repo (which its own worker fixes), never as a missing plan. Only a genuine planner-code failure is a no-op day, logged.

## Step 2 — dispatch the units, at their tier

Run each unit's **worker** at the tier its `smarts` names: a subagent on the matching capability tier for `high` / `medium` / `low`, or, for **`none`**, a **direct code / tool execution** with no subagent. Dispatch subagents with the **Agent/Task tool, never the multi-agent-orchestration Workflow tool** — its mandatory interactive opt-in stalls an unattended run on step one, waiting for an "Allow" no one is there to click. Each worker runs **exactly per its own doc**, handed the unit's `targets`; this routine adds **no** behavior. Read the worker doc from the right repo: a canon-pack unit's `worker` is a canon-relative path read from the canon checkout (`workerRepo` null). (A **local-pack** unit would carry `workerRepo` — the member repo — and a member-relative `worker` path, read from the member over MCP; that path only appears once the experimental `localTasksFor` seam is enabled in Step 1, which it is not by default.)

**Pass the plan's gathered enumeration through — don't make the worker re-fetch it.** The planner already enumerated each repo's touched branches / PRs / issues to build the unit (they're in the unit's `targets`, and the worker docs already say "the plan hands you the list — you don't enumerate"). So hand the worker that enumeration and let it start from there; **do not re-run `list_branches` / `list_pull_requests` / `list_issues` in the dispatch, and do not instruct the worker to re-enumerate** what the plan already holds — a worker fetches only the per-item detail it needs to verify (e.g. a branch's landed status against the default branch). Re-gathering the same lists the planner just fetched is pure duplicated cost and adds nothing; the plan is the single source of the worklist.

Concurrency — every unit is independent:

- **All units run concurrently**, capped to a sane batch. There is **no ordering and no barrier of any kind** — not per-repo, not fleet-wide. The repo tidy-up is a single `repo-tidy` unit whose worker does assess-branches → assess-PRs → triage-issues → reconcile-the-tracker in one pass, so that sequencing lives inside the one worker, never in the plan; nothing else waits on anything.
- **Await async downstream** — a unit that triggers a dispatch-only Action (a pack task's) is done only when that Action **completes**, not at the trigger; poll it (report at completion, not the trigger).

The subagent boundary delivers the guarantees the owner cares about: **failure isolation** (a unit that errors, stalls, or exits early fails *its own* subagent only), **context isolation**, and **behavior unchanged**. Also run the tidy tasks against the **canon repo** (it doesn't declare the tidy pack — its declaration carries only its home-only packs and the checks it runs on itself — but its PRs/branches/issues still need tending). Cap concurrency if the fleet is large, but **every** unit must be launched **and** waited on — a launched-but-unwaited unit is not a guaranteed unit. Wait for **everything** to settle before finishing.

## Step 3 — finalize migrations (retire)

Only after **every** unit has settled, run the migration **retire pass** ([../migrations/fleet-retire.mjs](../migrations/fleet-retire.mjs))'s `runRetirement(io, canonRepo, migrations, repos, unknownCount, today, appliedThisCycle)` over the **same** scoped repo list — it must be the COMPLETE managed set, since retirement fires only when the whole set is clean, so a missing repo could hide drift. Pass it Step 0's returned `appliedThisCycle` set directly (in memory) and the canon repo — the same repo the migrations were loaded from, where the deletes land. It probes each migration's fleet-wide completion via `io.read`, and for a migration the whole fleet has applied **and** that Step 0 touched on **no** repo this cycle (the **quiescence guard**: if that evidence is absent it retires **nothing**), it **stages the deletions** — the record and any files it relocated into the consumers — onto a stable retire branch and opens **one PR** (`io.ensureBranch` → `io.remove` on that branch → `io.openPr`, idempotent via `io.hasOpenPr`). Retirement is **irreversible**, so it is **never pushed to the canon's default branch**: the PR routes it through the canon's own CI (`barrier`, `reference-integrity`, the test suite), which validates that the deletion strands no inline reference before it can land — a clean retirement is a green, mergeable PR (never auto-merged); a stranding one is a red PR held for cleanup, so `main` never breaks. It runs **last**, on a fleet proven both converged and quiet, and it writes only to the canon repo. It is **pure decision logic (`smarts: none`)**.

## What the orchestrator itself must not do

The orchestrator is a sequencer, not a maintainer. It **never** decides the worklist by its own reasoning — the "should I run" gate is code (the planner); the orchestrator *runs* that code and reads its units, it doesn't reason its way to a plan. And it **never** runs a *worker's* logic, merges, commits docs by its own judgment, edits the canon, or writes to a step's tracking issue — every such action belongs to the worker (or the deterministic pass) that owns it. Running the plan's **code steps** — the planner and the migration apply/retire passes (all `smarts: none`, performing their reads/writes to members and the canon repo through the MCP `gh`) — is not the orchestrator reasoning; its *only* judgment-driven write is the failure log on its **own** tracking issue in the canon repo (next section). Keep it that thin.

## Tracking: log only failures, on the canon repo's own issue

This routine keeps its **own** standing tracking issue **in the canon repo**, separate from every step's — found **by title**, never a hard-coded number. Open it if it doesn't exist; reopen it if it was closed while a run still needs logging.

It logs **failures only** — the job is to guarantee every planned unit *ran* where it should, so the one thing worth surfacing is a run that **didn't** complete:

- **The planner run failed / the plan couldn't cover the fleet**, or **any worker failed, stalled, or exited without completing** (name the repo and task) → post a **dated comment** naming what was affected and the symptom.
- **Every planned unit completed where it should** (whether it changed anything or correctly did nothing) → **log nothing.** Workers already log their own *changes* to their own issues; a fleet-wide "all green" roll-up here would just be noise.

So on a normal day this routine writes nothing at all, and the only entries that accumulate are the failures you actually need to see.

## The launcher (Claude Code routine)

Keep the routine's config a **thin pointer** to this doc, not an inlined copy. Vendor this file (and the [fleet/](fleet/DESIGN.md) specs) in your canon repo, then schedule **this one routine** daily, pasting a prompt like the following and substituting the path where you vendored these docs:

> Run the fleet daily maintenance routine exactly as specified in `<path/to/routines/auto-all-repos-maintenance.md>`. Work **entirely through the GitHub MCP tools** — there is no `GITHUB_TOKEN` and no REST client. Back the code steps' injected I/O with those MCP tools: a read-only `gh(path)` for the planner (each path a `list_*`/`get_file_contents` call), and a semantic `io` object for the migration passes whose methods map 1:1 to the MCP tools (`io.read`→`get_file_contents`, `io.commit`→`push_files`, `io.remove`→`delete_file`, `io.ensureBranch`→`create_branch`, `io.hasOpenPr`→`list_pull_requests`, `io.openPr`→`create_pull_request`, `io.openIssue`→`issue_write`, `io.getDefaultBranch`→get repo). **The repos you manage are this session's granted scope** — you already know them, so do not enumerate GitHub. Name the **canon repo** (a known constant — the repo this doc and the migrations ship from). **Apply migrations**: run the apply pass's `applyToRepos(io, repos, migrations)` (migrations loaded via `migrations/registry.mjs`) to bring every managed repo to the canonical shape before any pack work — it lands each repo's pending writes as one commit honoring `push`/`pr` delivery, and **returns the ids it applied this cycle**; keep that set in hand for the retire step. **Run the planner**: run `planGivenRepos(gh, canonRepo, repos)` — it decides every unit in code (pack-agnostically; a broken pack is one isolated unit, not a missing plan); you do not decide the worklist by your own reasoning. (Do **not** pass a `localTasksFor` reader: local-pack daily tasks are an experimental, not-yet-enabled path — Step 1 — so leaving the seam unset plans only canon-pack tasks.) **Then dispatch each unit** as an isolated **plain subagent (never the Workflow orchestration tool, whose interactive opt-in an unattended routine can't answer)** at the tier its `smarts` names — or, for `smarts: none`, as a direct code/tool action, no subagent — running its `worker` doc (canon-relative, from the canon checkout) against its repo with the unit's `targets`. Every unit is independent: run them concurrently (capped to a sane batch) — there is no ordering and no barrier of any kind (the repo tidy-up is a single `repo-tidy` unit that sequences its own pass internally). Run the tidy task against the canon repo too. Await any dispatched Action to completion, not to its trigger. Once every unit has settled, **finalize migrations**: run `runRetirement(io, canonRepo, migrations, repos, unknownCount, today, appliedThisCycle)` with the same repo list and the applied-ids set from the apply step, to retire any migration the fleet has fully applied AND left untouched this cycle (it retires nothing if that evidence is absent). Retirement is **never pushed to the canon's default branch** — the pass stages its irreversible deletes onto a retire branch and opens one CI-gated PR (never auto-merged), so the canon's own checks validate reference-safety before it lands. Wait for everything to settle. You are a sequencer only: never decide the worklist by your own reasoning, never run a worker's logic, commit docs, edit the canon, or write to any step's tracking issue. Log **only failures** — the planner code failing, a plan that couldn't cover the reachable fleet, or a unit that didn't complete — to this routine's own standing tracking issue in the canon repo (found **by title**), naming the repo and task affected; on a clean day, log nothing.

Schedule it daily as a **Claude Code routine** (the Routines UI, or any scheduler that launches a Claude Code session — the steps need the session's MCP GitHub tools), from the canon repo. Every pack-task Action stays `workflow_dispatch`-only; this routine is the one schedule ([fleet/scheduling.md](fleet/scheduling.md)).

## Run on a capable model

The workers make **judgment calls** — each unit's `smarts` names the tier its worker needs. This routine's own step (run the planner, read its units, fan out at each unit's tier) is mechanical, but it drives those judgment-heavy workers, so run this routine — and therefore its subagents — on a capable model.

## What this routine must never do

- **Never decide the worklist by the orchestrator's own reasoning** — run the planner (the gate code) over the repos it was handed and read its units; the "should I run" gate is code, not a judgment call.
- **Never run a worker's logic itself** — it dispatches units into subagents (or runs the `none`-tier code action); it does not merge, commit, edit docs, or open any worker's output except its own failure log.
- **Never fan out through the Workflow tool** — dispatch runs as plain subagents (Agent/Task). Its mandatory interactive opt-in has no one to answer it in an unattended routine, so it stalls the whole run on step one.
- **Never let one unit's failure block another** — each is its own isolated subagent (or code action), and every unit is launched and waited on regardless of how the others fare.
- **Never artificially sequence units** — every unit is independent and runs concurrently; there is no ordering or barrier to impose.
- **Never also schedule the planner, a pack's task, or a per-repo maintenance routine** — this routine is their single schedule across the whole fleet.
- **Never log on a clean day** — it logs only failures to its own canon-repo tracking issue.
- **Never inline this spec, a worker's spec, or the plan into the launcher** — the launcher stays a thin pointer here.
