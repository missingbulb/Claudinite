# Google Identity (Sign-in-with-Google ID tokens)

Portable, project-agnostic practices for **validating Google Sign-In ID tokens on the server** — the
backend/validator side of Google Identity, true for any service that trusts a Google-issued OIDC ID
token (an API Gateway JWT authorizer, a hand-rolled `jsonwebtoken`/`jose` verifier, a Cloud Function).
Obtaining the token in the browser/extension client is a separate concern the `chrome-extension` pack
owns; these rules are about what the party that *checks* the token must get right.

- **Pin the validator's `audience` to your own OAuth client id — an empty/absent audience is an auth bypass.** A Google ID token is accepted on three things: a valid Google signature, the issuer `https://accounts.google.com`, and the `aud` (audience) claim. **Every** Google-issued ID token shares that one issuer, so signature + issuer alone prove only "some Google user signed in to *some* Google app" — the `aud` claim is the *only* thing that scopes a token to **your** app. A JWT authorizer or verifier configured with no audience (or an empty audience list) therefore accepts a token minted for *any* Google OAuth client while still passing signature and issuer checks — a full authentication bypass that looks completely valid. Always configure the validator's expected audience to your exact OAuth **Web application** client id; never leave it unset.
- **The OAuth client id is one value in two places, and the two must match exactly.** The client requests the ID token *for* a client id (that value becomes the token's `aud`), and the backend validator is configured to *expect* that same client id as its audience. When the two drift, every token is rejected with an opaque 401 even though it is well-formed and correctly signed — so the failure looks like a bug anywhere except the config that actually holds the mismatched literal. Source both from a single origin (one build-time constant, one deploy parameter) rather than two independently-edited copies, so they cannot diverge.
- **Trust the `email` claim only when `email_verified` is true.** A Google ID token carries an `email`, but it is trustworthy only in combination with `email_verified: true` — gate any privileged or identity-bearing action (posting, voting, moderation keyed on the address) on the verified flag, never on the mere presence of `email`. Practical note: behind an API Gateway JWT authorizer the boolean claim reaches your handler as the **string** `"true"`, not a JSON boolean, so compare against the string (or coerce) — a strict `=== true` silently rejects every genuinely-verified user.
